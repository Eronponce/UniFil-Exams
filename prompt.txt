Tarefa 7, corrigir PDF para que todas as provas tenham a mesma quantidade par de frentes

Problema

O sistema gerou um PDF com quantidade irregular de páginas. Isso quebra a lógica de impressão frente e verso e também quebra a lógica única de grampeamento.

O usuário quer mandar uma única regra de grampeamento para o lote inteiro.

Por isso, não basta cada prova ter número par de frentes. Todas as provas do mesmo lote precisam ter exatamente a mesma quantidade de frentes.

Regra nova

Cada prova individual deve ter a mesma quantidade de frentes.

Essa quantidade deve ser calculada assim.

1. Gerar ou simular todas as provas do lote.
2. Descobrir qual prova ficou com mais páginas.
3. Se a maior quantidade for ímpar, arredondar para o próximo número par mas lembrando de deixar o gabarito sempre na uyltima pagina.
4. Usar esse número como tamanho fixo de todas as provas do lote.
5. Completar as provas menores com páginas totalmente vazias até chegarem ao mesmo tamanho.

Exemplos

Se o Set A tiver 5 páginas, o tamanho padrão do lote vira 6 páginas.

Se o Set B tiver 4 páginas, ele também deve virar 6 páginas.

Se o Set C tiver 6 páginas, ele continua com 6 páginas.

Assim, todas as provas terão 6 frentes, ou seja, 3 folhas frente e verso.

Com isso, o usuário pode mandar uma única regra de grampeamento, a cada 3 folhas.

Outro exemplo

Se as provas tiverem estes tamanhos antes do ajuste.

Set A, 3 páginas.
Set B, 4 páginas.
Set C, 7 páginas.

A maior prova tem 7 páginas.

Como 7 é ímpar, o tamanho padrão vira 8 páginas.

Todas as provas devem ficar com 8 páginas.

Set A recebe 5 páginas vazias.
Set B recebe 4 páginas vazias.
Set C recebe 1 página vazia.

Comportamento esperado

1. Cada prova individual deve ser tratada como um bloco independente.
2. Todas as provas do mesmo lote devem ter a mesma quantidade de páginas.
3. Essa quantidade deve ser sempre par.
4. O tamanho fixo do lote deve ser baseado na maior prova gerada.
5. Provas menores devem receber páginas vazias no final.
6. A página vazia deve ser realmente vazia.
7. A página vazia não deve ter texto, cabeçalho, rodapé, logo ou numeração visível.
8. A próxima prova só pode começar depois do bloco fixo da prova anterior terminar.
9. Uma prova nunca pode começar no verso ou no espaço reservado da prova anterior.
10. Se houver cópias de sets, cada cópia também deve respeitar o mesmo tamanho fixo do lote.

Exemplo prático

Tenho 8 sets de prova.
Quero 2 cópias de cada set.
Isso gera 16 provas individuais.

O sistema deve calcular quantas páginas cada uma das 16 provas teria.

Depois deve encontrar a maior quantidade de páginas.

Se a maior tiver 5 páginas, o tamanho padrão do lote será 6 páginas.

Então todas as 16 provas devem ter 6 páginas.

Mesmo uma prova que originalmente teria 4 páginas deve receber 2 páginas vazias.

Assim, na impressão frente e verso, cada prova terá 3 folhas.

O grampeamento poderá ser feito sempre a cada 3 folhas.

Critérios de aceite

1. Todas as provas do lote devem ter a mesma quantidade de páginas.
2. Essa quantidade deve ser par.
3. O tamanho padrão deve ser calculado a partir da maior prova do lote.
4. Se a maior prova tiver número ímpar de páginas, arredondar para o próximo número par.
5. Provas menores devem receber páginas vazias até alcançar o tamanho padrão.
6. As páginas vazias devem ser totalmente vazias.
7. A regra deve funcionar para múltiplos sets.
8. A regra deve funcionar para múltiplas cópias de cada set.
9. A regra deve funcionar no PDF combinado.
10. Se existir PDF por set, ele também deve respeitar a mesma lógica quando fizer parte de um lote.
11. A imagem EvalBee e o gabarito devem respeitar essa separação.
12. O PDF final deve permitir uma única lógica de grampeamento para o lote inteiro.
13. O sistema deve evitar que provas tenham tamanhos diferentes dentro do mesmo lote.
Tarefa 8, criar fila para auditoria individual

Problema

Ao clicar várias vezes em auditar, a função pode bugar ou não concluir corretamente.

Comportamento esperado

A auditoria deve entrar em uma fila de tarefas.

Escopo

1. A auditoria é individual.
2. Não implementar auditoria em lote agora.
3. Cada tarefa audita uma questão.
4. A fila precisa continuar enquanto o servidor estiver rodando.
5. Não precisa sobreviver a reinício do servidor.

Critérios de aceite

1. Clicar em auditar cria uma tarefa na fila.
2. Cliques repetidos na mesma questão não criam tarefas duplicadas.
3. A tarefa deve continuar mesmo se o usuário sair da tela.
4. Ao voltar para a tela, o status deve aparecer atualizado.
5. Status esperados, aguardando, processando, concluída, erro, cancelada.
6. A fila deve ficar visível no topo da tela.
7. A fila pode aparecer como toast fixo, painel superior ou notificação global persistente.
8. O usuário deve poder cancelar tarefa.
9. Se a tarefa for cancelada antes de iniciar, ela não deve rodar.
10. Se a tarefa já estiver processando, marcar como cancelada e ignorar resultado se não for seguro interromper a chamada.
11. Se houver erro, salvar mensagem de erro e mostrar ao usuário.
12. Não deixar múltiplos cliques travarem a interface.
